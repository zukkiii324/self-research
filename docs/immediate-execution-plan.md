# Immediate Execution Plan

この文書は、プロジェクト全体の品質と効率を最大化するための施策を、`今すぐ着手できる実行単位` に落とした計画です。

## 実行方針

- `P0` と `P1` を先に潰す
- 実装、docs、テストを同時に更新する
- 抽象議論ではなく `完了条件` を先に決める
- 自動化は `止まらないこと` と `重複しないこと` を最優先にする

## フェーズ1: 今すぐ着手

### Task 1. 公開パイプラインを完全固定する

- 目的: 公開面の正系を `publish_site` に統一し、実運用とCIのズレをなくす
- 担当: 指揮担当 / 公開基盤
- 実行内容:
  - `deploy_static_blog.yml` を `publish_site` 前提で運用する
  - GitHub Pages の公開対象が `gh-pages` の `publish_site` 成果物であることを再確認する
  - README / docs / workflow の公開説明を `publish_site` に統一する
- 完了条件:
  - `content` 更新 -> `publish_site` 再生成 -> `gh-pages` 反映が1本の流れで説明できる
  - 文書内に `static_blog_site` を正系として説明する箇所が残っていない

### Task 2. 課題管理を公開判定の gate にする

- 目的: 品質未達のまま自動公開されることを防ぐ
- 担当: 指揮担当 / 編集長 / レビュー
- 実行内容:
  - `ISSUE_BOARD.md` の `P0` `P1` を run ごとに残す
  - `P0 Open` の場合は公開停止
  - `P1` が一定数以上未解消なら公開候補扱いに落とす
- 完了条件:
  - run ごとに課題管理表が生成される
  - 公開可否の判断ルールが docs に明文化されている

### Task 3. 重複判定を二段にする

- 目的: 焼き直し記事を減らす
- 担当: 自動化担当 / リサーチ
- 実行内容:
  - 候補記事と既存記事の比較対象を `title + summary` にする
  - 次段で `sections_to_cover` や brief 生成時の論点差分も見る
  - 重複度が高い場合は新規記事ではなく `既存記事更新候補` に振る
- 完了条件:
  - 類似記事候補が新規公開より先に除外または更新候補へ回る
  - 重複判定の基準値が docs に書かれている

### Task 4. 自動運転の fail-fast を徹底する

- 目的: 中途半端な実行失敗を減らす
- 担当: 自動化担当
- 実行内容:
  - `runner-command` 未設定時は即時失敗にする
  - secret 未設定、フィード空、公開面生成失敗の3ケースを明示的にログへ出す
  - 成功時と失敗時で state file のフォーマットを揃える
- 完了条件:
  - failure reason が `automation/runs` から追える
  - 無言失敗がない

### Task 5. モバイル品質の最低保証を入れる

- 目的: スマホ表示崩れの再発を防ぐ
- 担当: デザイナー / 公開基盤
- 実行内容:
  - `build_publish_site.py` の CSS にモバイル保護ルールを固定する
  - unit test で `overflow-wrap` `min-width: 0` `max-width: 100%` を検証する
  - 次段で実レンダリング検証を追加する
- 完了条件:
  - テストでモバイル保護ルールが落ちない
  - 主要テンプレートで横スクロールが発生しにくい

## フェーズ2: 今週中に進める

### Task 6. テーマ別 brief テンプレートを導入する

- 目的: 記事の論点ぶれを減らし、生成効率を上げる
- 担当: 指揮担当 / 戦略 / リサーチ
- 実行内容:
  - `AI活用` `Claude活用` `防災DX` で必須論点を分ける
  - `sections_to_cover` をテーマ別に固定する
  - 既存記事との差分を brief に強制埋め込みする
- 完了条件:
  - topic ごとに brief の型が違う
  - 記事の構造重複が減る

### Task 7. 参考リンクの命名規則を固定する

- 目的: 出典導線の読みやすさを上げる
- 担当: リサーチ / 最終編集
- 実行内容:
  - `組織名「文書名」` 形式を標準にする
  - URL裸出しを避ける
  - 参考名が空なら自動でURLを補完する
- 完了条件:
  - 参考セクションの見た目が記事間で揃う
  - 参考名を押すとリンク先へ飛べる

### Task 8. 記事スコアカードを導入する

- 目的: 良い記事と悪い記事の差を比較可能にする
- 担当: 指揮担当 / レビュー / 最終編集
- 実行内容:
  - run ごとに `正確性 / 独自性 / 可読性 / 公開完成度` を5段階で保存する
  - `RUN_SUMMARY.md` か別ファイルへ残す
- 完了条件:
  - run ごとに品質スコアが残る
  - 月次で比較できる

## フェーズ3: 次に着手する

### Task 9. 実レンダリングのモバイル検証を追加する

- 目的: CSS だけでなく見た目の崩れを検知する
- 担当: デザイナー / 公開基盤
- 実行内容:
  - iPhone 幅のトップ、カテゴリ、記事ページを自動キャプチャする
  - 主要要素のはみ出しや不可視をチェックする
- 完了条件:
  - CI で最低1本のモバイル画面検証が回る

### Task 10. 関連記事と更新導線を追加する

- 目的: 公開価値と回遊を上げる
- 担当: 戦略 / デザイナー
- 実行内容:
  - 各記事末尾に関連記事を出す
  - 同一テーマの更新版や比較対象へリンクする
- 完了条件:
  - カテゴリ内で回遊できる
  - 新規記事だけでなく既存資産も読まれる
