# Architecture

このプロジェクトは、単一の長文生成ではなく、役割分担された複数エージェントの連携で記事を作る前提です。

## 基本方針

- 役割ごとに責務を固定する
- 中間成果物をすべて保存する
- 最終版の前に必ずレビュー工程を入れる
- CLI は再実行しやすい単位で処理を分ける

## 実行モデル

現在のCLIは次の順番で工程を流します。

- `editor_in_chief`: 企画方針、品質基準、各工程への要求を定義する
- `strategy`: 誰に何をどう伝えるかを決める
- `research`: 必要な事実と根拠を整理する
- `outline`: 記事構成を作る
- `draft`: 本文を生成する
- `review`: 問題点と修正案を出す
- `final_edit`: 公開直前の最終稿に整える

## ディレクトリ責務

- `runs/`: 実行ごとの成果物置き場
- `docs/`: 設計と運用の説明
- `prompts/`: 各エージェントの指示文
- `examples/`: サンプル案件と実行例

## 再実行の考え方

現状のCLIは 1 回の実行で全工程を順に生成します。各工程の `prompt.md` と `response.md` が残るので、必要なら途中成果物を差し替えて再実行しやすい構造です。将来的に途中工程だけの再実行にも拡張しやすいよう、出力は工程別ディレクトリに分離しています。

## 品質の基準

- 事実関係が曖昧な箇所は残さない
- 見出しだけで論旨が追える
- Note 読者に向けて導入が軽すぎず重すぎない
- 文体が記事全体でぶれない
