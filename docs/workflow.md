# Workflow

## 1. 案件ブリーフを用意する

案件名、想定読者、伝えたい主張、参考資料、禁止事項をまとめます。

## 2. 実行する

CLI はブリーフを読み取り、案件用の `slug` を作成して `runs/<timestamp>-<slug>/` を初期化します。

## 3. 役割ごとの成果物を生成する

各工程は前の工程の出力を入力として使います。

```text
brief -> editor_in_chief -> strategy -> research -> outline -> draft -> review -> final_edit
```

## 4. レビューを反映する

レビューで出た問題点は、必要に応じて前工程に戻して修正します。特に、事実確認と構成の修正はドラフト修正より先に戻すほうが効率的です。

## 5. 公開前の最終版を出す

最終編集後の `FINAL_ARTICLE.md` を Note 投稿のベースにします。工程別の詳細は各ステージの `response.md` に残ります。

## 運用ルール

- 1 件 1 `run` にする
- 中間成果物を上書きせず残す
- 事実不明は断定しない
- 最終版だけを公開対象とする
