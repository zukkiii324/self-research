# Claude活用の評価を自動化する監視指標

4月6日は Claude を使い続けるためのモニタリング設計です。Claude 3 を定常的に使うなら、品質・コスト・ガバナンスの3軸で指標を作る必要があります。

### 品質

- 生成出力の `prompt_id` ごとに「承認」/「微修正」/「書き直し」に分類し、再プロンプトの発生率（リワーク率）を追う
- 「生成出力が実運用されたか」（例：Notionに反映されたか）を示すカバレッジ指標を作る
- 意図とずれていた場合は `intent_mismatch_reason` を5分類で定義し、カテゴリ別に分布を確認

### コスト

- Claude Kernel の `tokens` 数（input + output）を API 料金と照合し、週次でトークン単価を出す
- セッション再開回数などの実行回数をログし、1回の問いあたりの費用を算出
- 高コストなプロンプト（ex: 1回で50kトークン以上）には「出力概要のみ請求」形式の簡易プロセスを追加

### ガバナンス

- `response_format` で必ず JSON schema を使い、構造化チェックを通す
- フィードバックがあった場合は `feedback_ticket_id` を残し、誰がどこを修正したか記録
- 個人情報や機密が含まれる場合は `redaction_status` を自動判定し、OKになるまで送信を止める

### 今週の実施

1. 生成結果のリワーク率を Kibana/Looker で表示
2. 上位3つの高トークンプロンプトにコストアラートを張る
3. 個人情報検知のための軽量ルール（正規表現＋Claude自体の検知）を入れておく
