# Claudeのカーネル（Claude Kernel）で守る運用ルール

4月2日はClaude Kernel経由での開発・通知連携の整備を振り返ります。

Claude Kernelを使うと、ClaudeをCLIやCIに組み込めるので、「ログ、フィードバック、出力制御」をコードから管理できます。この日のポイントは次の通りです。

- **セッションの再現性**: Claude KernelはセッションIDと呼ばれる状態を使い、同じ履歴を再利用できます。自動ジョブから呼び出す際には毎回 `resume_session_id` を記録・再利用し、途中で構造が崩れないようにする。
- **ステートレス/ステートフルの境界**: 日次レポートなど結果だけ欲しい処理はステートレスで。継続的な質問応答はステートフルで保存しておくと、つぎの質問が途切れない。
- **出力の安定化**: Claudeは制約条件への準拠が強い。Kernelの `metadata` や `prompt_variables` にテンプレードを入れて、必ず「目的」「対象」「制約」を渡す。

## 今週の運用チェックリスト

1. Claude Kernelから呼び出したセッションIDを `sessions.json` に記録
2. 返却される出力を自動生成ログで `raw_response` と `parsed_response` で二重に保存
3. エラー時の再試行（API 502/504など）を2回以内に抑えるリトライロジックを入れる

このチェックを繰り返すと、Claude Kernelを単純なL2サービスではなく、社内ツールの基盤に育てられます。
